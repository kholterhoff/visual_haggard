# Static Publish Audit

This audit covers the current Rails app as the source system for a static public site to be hosted on GitHub Pages, with local export and Pagefind-based search.

## Goal

Keep Rails local for authoring and archive maintenance, but publish the public archive as static HTML/CSS/JS on GitHub Pages from the `gh-pages` branch.

## Current Public Surface

Public routes currently defined in [config/routes.rb](/Users/kateholterhoff/Documents/code/visual_haggard/config/routes.rb):

- `/`
- `/biography`
- `/editors_statement`
- `/novels`
- `/novels/:id`
- `/novels/:novel_id/editions`
- `/editions`
- `/editions/:id`
- `/editions/:edition_id/illustrations`
- `/illustrations`
- `/illustrations/:id`
- `/illustrators`
- `/illustrators/:id`
- `/blog`
- `/blog/:id`
- `/search`

Routes that are not part of the public static target:

- `/admin/*`
- `/up`

## What Is Already Static-Friendly

These parts of the current app are compatible with static export or close to it:

1. Public pages are mostly server-rendered ERB, not client-authenticated SPA views.
2. Most internal links already use path helpers and render as stable paths.
3. Legacy archive images already resolve to public URLs:
   - legacy S3 URLs from [app/models/edition.rb](/Users/kateholterhoff/Documents/code/visual_haggard/app/models/edition.rb)
   - legacy S3 URLs from [app/models/illustration.rb](/Users/kateholterhoff/Documents/code/visual_haggard/app/models/illustration.rb)
4. Homepage motion, PhotoSwipe, and other front-end behavior are client-side and can work on a static host.
5. The site does not require public write operations.

## Confirmed Static-Hosting Blockers

### 1. Search is fully server-side

Current implementation:

- [app/controllers/search_controller.rb](/Users/kateholterhoff/Documents/code/visual_haggard/app/controllers/search_controller.rb)
- [app/services/archive_search.rb](/Users/kateholterhoff/Documents/code/visual_haggard/app/services/archive_search.rb)
- [app/views/search/index.html.erb](/Users/kateholterhoff/Documents/code/visual_haggard/app/views/search/index.html.erb)
- header form in [app/views/layouts/application.html.erb](/Users/kateholterhoff/Documents/code/visual_haggard/app/views/layouts/application.html.erb)

Why this blocks static hosting:

- search results are generated by live SQL queries
- illustration pagination on search depends on `params[:page]`
- GitHub Pages cannot run Rails requests

Required change:

- replace live `/search` with a static search page backed by Pagefind

### 2. Novel pagination uses query-string pages

Current implementation:

- [app/controllers/novels_controller.rb](/Users/kateholterhoff/Documents/code/visual_haggard/app/controllers/novels_controller.rb)
- [app/views/novels/index.html.erb](/Users/kateholterhoff/Documents/code/visual_haggard/app/views/novels/index.html.erb)

Why this blocks static hosting:

- the novels index uses `?page=2`, `?page=3`, etc.
- GitHub Pages serves files, not server-side query variants
- `/novels?page=2` and `/novels?page=3` would both resolve to the same `novels/index.html`

Required change:

- export paginated novels to path-based pages such as `/novels/page/2/`
- or replace server pagination with client-side pagination/filtering on a static dataset

Recommended choice:

- path-based static pagination for the novels archive

### 3. Newly uploaded images are not guaranteed to be static-safe

Current implementation:

- attachment URL helper in [app/helpers/application_helper.rb](/Users/kateholterhoff/Documents/code/visual_haggard/app/helpers/application_helper.rb)
- Active Storage config in [config/storage.yml](/Users/kateholterhoff/Documents/code/visual_haggard/config/storage.yml)
- model display helpers in:
  - [app/models/edition.rb](/Users/kateholterhoff/Documents/code/visual_haggard/app/models/edition.rb)
  - [app/models/illustration.rb](/Users/kateholterhoff/Documents/code/visual_haggard/app/models/illustration.rb)

Why this blocks static hosting:

- if `display_cover_source` or `display_image_source` returns an attached blob, the published site will reference `/rails/active_storage/...`
- GitHub Pages cannot serve Rails blob routes

Required change:

- define a publishing rule for attached images

Recommended choice:

- do not rely on Rails blob URLs for published pages
- before publish, either:
  - upload new images to durable public object storage and store public URLs, or
  - copy attachments into the static export and rewrite their URLs

The first option is operationally cleaner.

### 4. GitHub Pages base path must be decided now

Current implementation:

- internal links are rendered as root-relative paths such as `/novels`, `/illustrators/24`, `/search`

Why this matters:

- root-relative links work correctly on:
  - a custom domain
  - a GitHub Pages user site at the root
- they do not work correctly on a GitHub Pages project site under `/<repo>/...`

Required change:

- either use a custom domain
- or add a base-path helper for all generated URLs

Recommended choice:

- use a custom domain so the current path structure can stay root-relative

### 5. Public blog routes are not implemented

Current implementation:

- [app/controllers/blog_posts_controller.rb](/Users/kateholterhoff/Documents/code/visual_haggard/app/controllers/blog_posts_controller.rb)
- [app/views/blog_posts/index.html.erb](/Users/kateholterhoff/Documents/code/visual_haggard/app/views/blog_posts/index.html.erb)
- [app/views/blog_posts/show.html.erb](/Users/kateholterhoff/Documents/code/visual_haggard/app/views/blog_posts/show.html.erb)

Why this matters:

- these routes are public
- but the views are still placeholders
- exporting them as-is would publish broken/unfinished pages

Required change:

- either implement them
- or exclude/remove public blog routes before the static launch

Recommended choice:

- exclude them from the first static publish unless they are intentionally part of the public archive

## Export Scope Recommended For First Static Launch

Export these pages:

- `/`
- `/biography`
- `/editors_statement`
- `/novels`
- static path-based paginated novel pages
- `/novels/:id`
- `/editions/:id`
- `/illustrations`
- `/illustrations/:id`
- `/illustrators`
- `/illustrators/:id`
- `/search` as a static shell page for Pagefind
- `404.html`

Exclude initially:

- `/blog`
- `/blog/:id`
- `/editions`
- nested duplicate index routes unless there is a clear public need for them

## Export Strategy Recommended

Add a local task:

- `bin/rails static:export`

Recommended behavior:

1. render each public route to HTML
2. write files into `dist/`
3. emit path-based novel pagination
4. copy any static-safe local assets needed for Pages
5. leave search results to Pagefind indexing

Recommended output shape:

- `dist/index.html`
- `dist/biography/index.html`
- `dist/editors_statement/index.html`
- `dist/novels/index.html`
- `dist/novels/page/2/index.html`
- `dist/novels/59/index.html`
- `dist/editions/111/index.html`
- `dist/illustrations/962/index.html`
- `dist/illustrators/24/index.html`
- `dist/search/index.html`
- `dist/404.html`

## Pagefind Integration Plan

Use Pagefind against `dist/` after export.

Required work:

1. create a static search shell page
2. replace the current GET search form behavior on the published site with Pagefind JS
3. ensure tags, titles, and relevant descriptive text are present in exported HTML so Pagefind can index them

Recommended local publish sequence:

1. `bin/rails static:export`
2. `pagefind --site dist`
3. sync `dist/` into the `gh-pages` worktree
4. commit and push `gh-pages`

## Decisions Still Needed

1. Will the published site use a custom domain?
2. Should public blog pages ship in the first static version, or be removed for now?
3. For newly uploaded images, do we want:
   - public object storage URLs, or
   - copied image files inside the static export?

## Recommended Implementation Order

1. Add static export task and route inventory
2. Convert novel pagination to static path-based pages
3. Replace server search with Pagefind
4. Define and implement the image publishing rule for new uploads
5. Add local publish script and `gh-pages` workflow
6. Optionally add custom domain support
